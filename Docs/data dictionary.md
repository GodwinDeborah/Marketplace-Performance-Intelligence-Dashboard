# Marketplace Performance Intelligence Dashboard
# Data Dictionary

## Overview

This document describes the data model used in the Marketplace Performance Intelligence Dashboard.

The solution follows a **Star Schema** consisting of one fact table and five dimension tables. The model was designed to optimize analytical queries, simplify report development, and improve dashboard performance in Power BI.

---

# Data Model

## Fact Table

| Table Name | Description |
|------------|-------------|
| fact_table_orders | Stores transactional marketplace order data and serves as the central fact table. |

---

## Dimension Tables

| Table Name | Description |
|------------|-------------|
| dim_date | Calendar dimension used for time-based analysis. |
| dim_restaurant | Restaurant information and descriptive attributes. |
| dim_category | Food category information. |
| dim_location | Geographic information including city and state. |
| dim_dish | Dish information available across restaurants. |

---

# Fact Table

## fact_table_orders

### Description

Contains transactional order records for all marketplace purchases.

Each record represents one customer order.

### Columns

| Column | Data Type | Description |
|---------|-----------|-------------|
| order_id | Integer | Unique identifier for each order. |
| restaurant_id | Integer | Foreign key to the restaurant dimension. |
| category_id | Integer | Foreign key to the category dimension. |
| dish_id | Integer | Foreign key to the dish dimension. |
| location_id | Integer | Foreign key to the location dimension. |
| date_id | Integer | Foreign key to the date dimension. |
| price | Decimal | Revenue generated by the order. |
| rating | Decimal | Customer rating for the order. |

# Relationships

The dashboard follows a one-to-many relationship model.

| From | To | Relationship |
|------|----|--------------|
| dim_date[date_id] | fact_table_orders[date_id] | One-to-Many |
| dim_restaurant[restaurant_id] | fact_table_orders[restaurant_id] | One-to-Many |
| dim_category[category_id] | fact_table_orders[category_id] | One-to-Many |
| dim_location[location_id] | fact_table_orders[location_id] | One-to-Many |
| dim_dish[dish_id] | fact_table_orders[dish_id] | One-to-Many |

Cross-filter direction is configured as **Single** from dimensions to the fact table.

---

# Hidden Columns

The following technical columns are hidden from report users to simplify the data model:

- order_id
- restaurant_id
- category_id
- dish_id
- location_id
- date_id

These fields remain available for relationships but are excluded from the reporting interface.

---

# Measures

Business calculations are implemented using DAX measures rather than calculated columns where appropriate.

Key measures include:

- Total Revenue
- Total Orders
- Average Order Value
- Average Rating
- Restaurant Revenue
- Revenue Contribution %
- Top 10 Revenue Share
- Active Restaurants
- Active Cities
- Distinct Categories
- Distinct Dishes

Refer to **DAX calculations.md** for the complete list of measures.

---

# Data Modeling Approach

The solution uses a **Star Schema** because it provides:

- Improved query performance
- Simplified DAX calculations
- Easier report maintenance
- Reduced data redundancy
- Better scalability

Dimension tables contain descriptive attributes, while the fact table stores measurable business events.

# Naming Conventions

The following naming conventions were used throughout the project:

### Fact Tables

Prefix:

```
fact_
```

Example:

```
fact_table_orders
```

### Dimension Tables

Prefix:

```
dim_
```

Examples:

```
dim_restaurant
dim_location
dim_category
```

### Measures

Business-friendly naming was used for all DAX measures.

Examples:

- Total Revenue
- Average Rating
- Active Restaurants
- Top 10 Revenue Share

This improves report readability and usability for business users.

# Summary

The Marketplace Performance Intelligence Dashboard is built on a clean dimensional model designed according to data warehousing best practices.

The combination of a centralized fact table, conformed dimensions, hidden technical fields, and reusable DAX measures provides a scalable foundation for interactive business intelligence reporting.
